01 · The Opportunity
Every JS build tool's cache is lying to you.
You declared the inputs. You trusted they were right.
rage observes them instead.
01

The cache confidence gap

Turborepo, Nx, and lage all cache based on declared input lists in config files. When those declarations drift from reality — and they do — you get stale outputs. Silently. rage's sandbox observes what files a build actually touches, then fingerprints those. You can't misconfigure what you don't configure.

02

Bazel correctness is real but brutal

Bazel's sandbox model solves this problem — but it costs you 2–4 months of ramp-up, BUILD file maintenance, Starlark mastery, and a full-time build engineer. The insight isn't "Bazel is wrong." It's "nobody has made Bazel's correctness model accessible to JS developers without the BUILD tax."

03

The quadrant is genuinely empty

Simple + Correct does not exist in this market. Turborepo owns Simple + Trust-me. Bazel/Buck2 own Complex + Correct. rage targets the top-right quadrant that no tool has claimed. That's the white space.

04 · Core Messages

The three things we say

Every developer conversation leads with one of these. Test with engineers before landing on copy.

🔬

Correct by construction, not by configuration

rage doesn't ask you to declare what files your build reads. It watches what your build actually reads — using a macOS DYLD interposer that hooks every open(), stat(), and rename() syscall. The pathset becomes the fingerprint. A cache hit means every byte your build touched is identical. Not "we think so." Proven.

Differentiator vs: Turborepo · Nx · lage · Buck2
🏗️

BuildXL-grade correctness. Zero BUILD files.

BuildXL ships Windows. It solves this exact problem at Microsoft-internal scale using the same two-phase WF→SF fingerprint algorithm rage implements. The difference: BuildXL requires DScript declarations. rage's TypeScript plugin infers declared inputs automatically, and the sandbox validates them. You get the algorithm without the authoring tax.

Differentiator vs: Bazel · Buck2 · Aspect Build
⚡

The daemon that stays converged

Nx has a daemon for performance. rage's daemon maintains desired state. Tell it "I want build to be green across all packages." It converges. File changes trigger reconvergence. It's not a cache proxy — it's a controller with a reconciliation loop. The status page is a live view of convergence state via WebSocket push.

Differentiator vs: Every tool in the market

08 · COE Challenges

The hard questions

The skeptic in the room will ask these. Expand each for the prepared answer.

⚠ "You're JS-only and macOS-only. Turborepo works everywhere. Why would anyone switch?"
Correct. That's why we don't claim SOTA today. At Gate 0, rage is an alpha for developers who've hit the "cache lied to me" problem on macOS TypeScript repos. The narrative is "here's the architecture and why it's right" — not "ship it to production now." Gate 1 (Linux sandbox + remote cache) is when we have a serious answer to this question. We should be explicit about this timeline, not defensive about the current state.
⚠ "macOS 26 (Tahoe) strips DYLD_INSERT_LIBRARIES from system binaries. Your sandbox story breaks."
Known and documented in the codebase (sandbox/src/macos.rs). Hardened system binaries on macOS 26+ evade DYLD injection — specifically /bin/sh, /usr/bin/node if system-installed, and similar. Rage works around this by preferring Homebrew-installed interpreters, which are ad-hoc signed and injectable. The limitation: any task that spawns a system-hardened binary loses interposition for that subtree. Affects a subset of builds, not all. The Linux eBPF sandbox (Gate 1) closes this gap cleanly — eBPF doesn't have this limitation. We should document the Tahoe caveat clearly on rage.build.
⚠ "No benchmarks. You can't claim 'fast like BuildXL' without numbers."
100% correct. We don't have benchmarks. We have architectural claims: blake3 hashing, Tokio wave-parallel execution, WF lookup that only hashes declared globs (no sandbox execution on cache hit). These are sound but unmeasured. The SOTA gate (Gate 3, Q1 2027) requires a published, reproducible benchmark suite. Until then, we should not use the word "fastest" or imply performance superiority. The correctness story stands on its own without performance claims.
⚠ "Buck2 is also Rust-based and also correct. What's actually different?"
Buck2 still requires BUILD file declarations (Starlark). You author your inputs. If your declarations drift, your cache drifts. rage observes inputs — no authoring required. That's the fundamental difference. Buck2 also has zero managed remote cache service, tiny community outside Meta, and adoption that's essentially stalled since the 2022 announcement. We're not competing with Buck2 for the same audience — Buck2's users are large polyglot shops that already own BUILD file expertise. rage's users are JS developers who want correctness without the maintenance burden.
⚠ "No remote cache means no team workflow. You're a developer toy."
Yes, until Gate 1. This is the honest answer. The alpha positioning should be "correctness engine for developers evaluating the architecture" not "replace your CI today." The remote cache backend (S3/Azure Blob, open protocol — not a Vercel or Nx lock-in) is the feature that turns rage into a team tool. It's planned and architecturally obvious to implement given the existing TwoPhaseCache abstraction. Gate 1 is the team-readiness milestone. Until then, the target audience is build-system-curious developers and teams doing evaluation, not production deployments.
⚠ "Nx already has 20K GitHub stars and enterprise sales. You're late."
Late to the JS task runner market, yes. Not late to the correctness market — that doesn't exist yet. The positioning isn't "better Nx." It's "the tool that solves the problem Nx can't solve without BUILD file maintenance and vendor lock-in." The lage installed base (existing Microsoft ecosystem users) is the first-mover wedge. From there, the expansion story is teams that hit Nx Cloud pricing friction or want correctness guarantees. We don't need to replace Nx. We need to own the team that evaluated Nx and wanted something more trustworthy.
